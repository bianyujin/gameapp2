# Firebase 流量优化方案

## 为什么一天用了107MB？

### 原因分析：

1. **全量下载** - 每次同步都下载整个 `/games.json`
   - 假设1000个游戏，每个5KB，就是5MB
   - 20个人同步就是100MB

2. **没有缓存** - 数据没变也重新下载

3. **重复请求** - 配置读取、游戏数据读取都是独立请求

---

## 已做的优化 ✅

### 1. ETag 缓存
- 利用 Firebase 的 ETag 响应头
- 数据没变时返回 304，不下载内容
- 可节省 90%+ 的流量

### 2. 配置缓存
- `/config.json` 也用同样的缓存机制
- 版本号、密码等配置不会重复下载

---

## 更多优化建议

### 方案一：数据分片（推荐）
把数据按分类或首字母分片：

```
/games/a/...
/games/b/...
/games/c/...
```

只下载用户看的分类。

### 方案二：静态 JSON + GitHub Pages（最省流量）
1. 把游戏数据打包成 `games.json`
2. 上传到 GitHub Releases 或 CDN
3. Firebase 只存：
   - `/config.json`（版本、密码）
   - `/data_version.json`（数据版本号）

**流量对比：**
| 方案 | 100人同步 |
|------|-----------|
| 当前方案 | ~500MB |
| 静态JSON | ~5MB（只下载更新） |

### 方案三：压缩数据
用 gzip 压缩，JSON 压缩率约 70-80%。

## 当前方案的用户数估算

假设：
- 每个游戏数据：5KB
- 总游戏数：1000
- 每次同步：5MB
- Firebase 免费额度：360MB/天

| 用户数/天 | 流量 | 状态 |
|-----------|------|------|
| 20人 | 100MB | ✅ 安全 |
| 50人 | 250MB | ⚠️ 注意 |
| 70人 | 350MB | 🟠 接近上限 |
| 100人 | 500MB | ❌ 超支 |

**加上 ETag 缓存后（假设50%命中率）：**
| 用户数/天 | 流量 |
|-----------|------|
| 40人 | 100MB |
| 100人 | 250MB |
| 140人 | 350MB |
